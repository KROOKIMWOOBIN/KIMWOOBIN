## 1. 계층형 아키텍처 (Layered Architecture)

### MVC 기준 구조 흐름
```text
사용자 / 브라우저
  ↓
Controller
요청을 받는 곳
  ↓
Service
실제 기능을 처리하는 곳
  ↓
Model / Domain
비즈니스 규칙이 있는 곳
  ↓
Repository
DB와 연결하는 곳
  ↓
DB
```

### 왜 쓰는가?
계층으로 분리하여, 내가 지금 어디 영역을 수정하는 지 관심사 분리하기 위해 존재

### 특징
1. 상하 관계의 의존성
2. 레이어 간 격리

### 언제 쓰는가?
시스템의 규모가 크지 않고 비즈니스 로직이 비교적 단순할 때.

### 장단점
1. 구조가 단순하고, 진입 장벽이 낮음 -> 즉, 개발 난이도 낮다.
2. DB랑 서비스가 너무 의존 관계가 높아, 단위 테스트하기 어려워짐 
    - 해결 방안 : 서비스 로직을 도메인 객체 내부에 설계하여 역할 부담을 하면 단위 테스트하기 쉬워짐

## 2. 헥사고날 아키텍처 (Hexagonal / Ports and Adapters)

### MVC 기준 구조 흐름
```text
사용자 / 브라우저
  ↓
Controller
외부 요청을 받는 입구
  ↓
Service Interface
Controller가 호출할 기능 약속
  ↓
Service / Domain
핵심 비즈니스 로직
  ↓
Repository Interface
DB 저장 방식을 숨기는 약속
  ↓
Repository 구현체 / 외부 API 호출 코드
실제 연결 코드
  ↓
DB / 외부 API / 메시지 큐
```

### 왜 쓰는가?
비즈니스 로직을 외부 시스템 및 기술(클라이언트, DB, 외부 API 등) 인프라로부터 완전히 분리하기 위해 존재.

### 특징
1. Port : 서비스가 DB 영역을 향해 "나한테 데이터 주려면 이 모양(도메인 객체)으로 맞춰서 가져와" 하고 레포지토리 인터페이스를 뚫어놓은 것.
2. Adapter : 실제 JPA나 SQL을 써서 DB 엔티티(UserJpaEntity)를 조회한 다음, 서비스 구멍이 요구하는 순수 도메인 객체 모양으로 변환해서 납품하는 클래스.

### 언제 쓰는가?
비즈니스 로직이 복잡하고, 외부 시스템이 자주 바뀌거나 테스트 용이성이 중요할 때

### 장단점
- 도메인이 인프라에 독립적 -> 단위 테스트가 쉬움
- 파일 클래스가 많아지고 구조가 복잡해짐 (Request, Response, Entity, Info(DTO))등 전부 구현해야 하고, 매번 변환해 주어야 함

## 3. 어니언 아키텍처 (Onion Architecture)

### MVC 기준 구조 흐름
```text
Controller / Repository / Spring / DB 연결 코드
가장 바깥쪽 코드
  ↓
Service
사용자의 요청을 하나의 기능으로 처리
  ↓
Domain Service
복잡한 비즈니스 규칙 처리
  ↓
Domain Model
가장 중요한 비즈니스 데이터와 규칙

핵심: Model / Domain을 가장 중심에 둔다
의존성 방향: 바깥쪽 코드 -> 안쪽 도메인
```

### 왜 쓰는가?
도메인 모델을 시스템의 중심에 놓고, 모든 의존성이 바깥에서 안쪽으로만 향하도록 강제하기 위해 존재

### 특징
1. 동심원 구조 (안 -> 바깥 순서)
2. 의존성 방향이 항상 바깥에서 안쪽으로

### 언제 쓰는가?
도메인 로직이 매우 복잡하고, DDD 와 함께 사용할 때, 장기적으로 유지보수할 대규모 시스템에 적합

### 장단점
- DDD 개념을 자연스럽게 녹일 수 있음
- 가장 복잡한 구조, 파일 수와 추상화 레벨이 최고 수준

## 4. CQRS 아키텍처 (Command Query Responsibility Segregation)

### MVC 기준 구조 흐름
```text
일반 MVC라면

게시글 등록 / 수정 / 삭제
게시글 목록 / 상세 조회
  ↓
PostController
  ↓
PostService
  ↓
PostRepository
  ↓
Post DB


CQRS를 적용하면

[쓰기 전용 흐름]
게시글 등록 / 수정 / 삭제
  ↓
PostCommandController
  ↓
PostCommandService
  ↓
Write DB
원본 데이터를 정확하게 저장

Write DB
  ↓
조회하기 좋은 형태로 데이터 복사
  ↓
Read DB

[읽기 전용 흐름]
게시글 목록 / 상세 조회
  ↓
PostQueryController
  ↓
PostQueryService
  ↓
Read DB
화면에 보여주기 좋게 저장된 데이터 조회
```

```text
핵심

Command = 데이터 변경
Query   = 데이터 조회

즉, CQRS는
"저장하는 코드"와 "조회하는 코드"를 분리하는 방식이다.
```

### 왜 쓰는가?
쓰기와 읽기의 요구사항이 본질적으로 다르다는 인식에서 출발

### 특징
1. 쓰기 최적화 / 읽기 최적화 따로 존재

### 언제 쓰는가?
- 읽기/쓰기 비율이 극단적으로 다를 때 (e.g., 읽기: 99% : 쓰기 1%)

### 장단점
- 읽기/쓰기 각각 독립적으로 스케일링 가능
- 최종 일관성 문제 : 쓰기 후 읽기 DB에 즉시 반영 안 될 수 있음
